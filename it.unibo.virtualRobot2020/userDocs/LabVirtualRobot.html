INTRODUCTION

Our company (Unibo) has developed a virtual environment (named WEnv) that includes a virtual robot.
The virtual environment has been mainly built by Pierfrancesco Soffritti using the three.js JavaScript library (see ConfigurableThreejsApp ) .

Starting

The reference project for this introduction is it.unibo.robots19.
  1. Install Node.js
  2. Open a terminal on it.unibo.robots19\node\WEnv\server and execute 
    npm install
  3. In the directory it.unibo.robots19\node\WEnv\WebGLScene execute 
    npm install
  4. In the directory it.unibo.robots19\node\WEnv\server\src execute 
    node main 8999
  5. Open a browser on 
    localhost:8090

INTERACTION

The virtual robot uses the Node library https://github.com/einaros/ws to accept commands sent on a websocket at port 8091. 
 {"robotmove":"MOVE", "time":T} 	with MOVE = turnLeft | turnRight | moveForward | moveBackward | alarm, T = naturalNum
These commands are redirected (using socket.io) to the WebGLScene.

Moreover, the virtual environment sends to the clients connected to the ws port 8091:
  • Data emitted by the sonars included in the scene when they detect some object (the moving robot)
  • Data emitted by the impact sensors put in front and in the rear of the robot, when they detect an obstacle (fixed or mobile). For example: 
    { "sonarName": "sonarName", "distance": 1, "axis": "x" }

{ "collision" : "false", "move": "moveForward"}

The class AppMsg.kt implements our idea of messages at application-level. The external representation is a String written in Prolog (JSON-compatible) syntax as a Prolog-Term with 5 arguments: 
msg( MSGID, MSGTYPE, SENDER, RECEIVER, CONTENT, SEQNUM )
We distinguish among different application-message types: 
enum class AppMsgType{ event, dispatch, request, reply, invitation }

Clients as FSM

WalkerHttpCaller.kt

Request-response

The behavior of the WalkerHttpCaller.kt is structured as a Finite State MAchine (FSM) that moves the virtual robot ahead (by using the WEnvHTTPSupport.kt) until an obstacle (a wall) is found.
If the robot starts at its HOME cell (0,0), the result could be the map created by the WEnvHTTPSupport.kt: 
|1, 
|1, 
|1, 
|r, 
|X, 
 direction=DOWN
walker.kt

Event-based

The behavior of the walker.kt is structured as a Finite State MAchine (FSM) that moves the virtual robot ahead (by using the WEnvConnSupport.kt) until an obstacle (a wall) is found.
The result of the computation could be the number of steps done. A room map can be built by the application.

The events raised from WEnv ( endmove, collision, sonarName ) are mapped into dispatches sent to walker.

We could set the step time so that (e.g. 400) each step will cover a distance equal to the length of the robot.
